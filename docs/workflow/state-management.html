State Management

Aziron's workflow engine provides comprehensive state management capabilities to ensure reliable execution, recovery from failures, and persistence across system restarts.

Overview‚Äã

The state management system handles:

  • Execution State: Current workflow execution status and progress
  • Context Persistence: Workflow variables and intermediate results
  • Recovery Mechanisms: Resume workflows after failures or restarts
  • History Tracking: Complete audit trail of workflow executions
  • Checkpoint Management: Save and restore execution checkpoints

Execution States‚Äã

Workflow Execution States‚Äã

StateDescriptionTransitions
pendingWorkflow queued for execution‚Üí running, cancelled
runningCurrently executing‚Üí completed, failed, paused
pausedTemporarily suspended‚Üí running, cancelled
completedSuccessfully finishedTerminal state
failedExecution failed‚Üí retrying, Terminal
cancelledManually cancelledTerminal state
retryingAttempting retry‚Üí running, failed

Step Execution States‚Äã

StateDescriptionCan Retry
pendingWaiting to executeN/A
runningCurrently executingNo
completedSuccessfully finishedNo
failedExecution failedYes
skippedConditionally skippedNo
timeoutExecution timed outYes

Core Properties‚Äã

type WorkflowExecution struct {
    ID                string                    `json:"id"`
    WorkflowID        string                    `json:"workflow_id"`
    Status            WorkflowStatus            `json:"status"`
    Context           map[string]interface{}    `json:"context"`
    Results           map[string]interface{}    `json:"results"`
    StepStates        map[string]StepExecution  `json:"step_states"`

    CompletedAt       *time.Time                `json:"completed_at,omitempty"`
    StartedBy         string                    `json:"started_by"`
    Error             string                    `json:"error,omitempty"`
    RetryCount        int                       `json:"retry_count"`
    Checkpoints       []ExecutionCheckpoint     `json:"checkpoints"`
}

Variable Scoping‚Äã

Variables in workflow context follow a scoping hierarchy:

  1. Global Context: Available to all steps
  2. Step Results: Results from previous steps
  3. Local Variables: Step-specific variables
  4. System Variables: Built-in system values

Variable Substitution‚Äã

{
  "content": "Hello {{user.name}}, your order {{order.id}} status is {{step_check_status_result.status}}"
}

Available Variables:

  • {{variable_name}} - Global context variable
  • {{step_<step_id>_result}} - Result from specific step
  • {{step_<step_id>_<property>}} - Property from step result
  • {{system.timestamp}} - Current timestamp
  • {{system.execution_id}} - Current execution ID
  • {{system.workflow_id}} - Current workflow ID

Persistence‚Äã

Automatic Persistence‚Äã

The workflow engine automatically persists state at key points:

  • Workflow Start: Initial state and context
  • Step Completion: Step results and updated context
  • Step Failure: Error information and retry state
  • Workflow Pause: Complete execution state
  • Workflow Completion: Final results and metrics

Database Schema‚Äã

-- Workflow executions table
CREATE TABLE workflow_executions (
    id UUID PRIMARY KEY,
    workflow_id VARCHAR(255) NOT NULL,
    status VARCHAR(50) NOT NULL,
    context JSONB NOT NULL,
    results JSONB,
    started_at TIMESTAMP NOT NULL,
    completed_at TIMESTAMP,
    started_by VARCHAR(255) NOT NULL,
    error TEXT,
    retry_count INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Step executions table
CREATE TABLE step_executions (
    id UUID PRIMARY KEY,
    execution_id UUID REFERENCES workflow_executions(id),
    step_id VARCHAR(255) NOT NULL,
    status VARCHAR(50) NOT NULL,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    duration BIGINT, -- microseconds
    result JSONB,
    error TEXT,
    retry_count INTEGER DEFAULT 0,
    metadata JSONB
);

-- Execution checkpoints table
CREATE TABLE execution_checkpoints (
    id UUID PRIMARY KEY,
    execution_id UUID REFERENCES workflow_executions(id),
    checkpoint_name VARCHAR(255) NOT NULL,
    state_data JSONB NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

Checkpoints‚Äã

Automatic Checkpoints‚Äã

The system automatically creates checkpoints at strategic points:

type ExecutionCheckpoint struct {
    ID             string                 `json:"id"`
    ExecutionID    string                 `json:"execution_id"`
    CheckpointName string                 `json:"checkpoint_name"`
    StateData      map[string]interface{} `json:"state_data"`
    CreatedAt      time.Time              `json:"created_at"`
}

Checkpoint Types‚Äã

  • step_start: Before step execution begins
  • step_complete: After successful step completion
  • branch_split: Before parallel execution split
  • branch_merge: After parallel execution merge
  • error_recovery: Before error recovery attempt
  • manual: User-triggered checkpoint

Creating Manual Checkpoints‚Äã

{
  "id": "create_checkpoint",
  "type": "checkpoint",
  "config": {
    "checkpoint_name": "before_expensive_operation",
    "include_context": true,
    "include_results": true
  }
}

Recovery Mechanisms‚Äã

Automatic Recovery‚Äã

The workflow engine provides several recovery mechanisms:

1. Step Retry‚Äã

{
  "id": "resilient_step",
  "type": "message",
  "config": {
    "retry_policy": {
      "max_retries": 3,
      "retry_delay": 1000,
      "backoff_multiplier": 2.0,
      "retry_on": ["timeout", "network_error", "rate_limit"]
    }
  }
}

2. Workflow Resume‚Äã

// Resume workflow from last checkpoint
execution, err := workflowEngine.ResumeExecution(executionID)
if err != nil {
    log.Printf("Failed to resume execution: %v", err)
    return err
}

3. State Restoration‚Äã

// Restore execution state from checkpoint
err := workflowEngine.RestoreFromCheckpoint(executionID, checkpointName)
if err != nil {
    log.Printf("Failed to restore from checkpoint: %v", err)
    return err
}

Recovery Strategies‚Äã

Optimistic Recovery‚Äã

Best Practices‚Äã

State Design‚Äã

  1. Keep Context Minimal: Only store necessary data in context
  2. Use Immutable Data: Avoid modifying shared context directly
  3. Clear Variable Names: Use descriptive names for context variables
  4. Version Context: Include version information for context evolution

Error Handling‚Äã

  1. Graceful Degradation: Design workflows to handle partial failures
  2. Meaningful Errors: Provide clear error messages and recovery hints
  3. Error Classification: Categorize errors for appropriate handling
  4. Recovery Testing: Test recovery mechanisms regularly

Performance Optimization‚Äã

  1. Checkpoint Frequency: Balance between recovery speed and overhead
  2. Context Size: Monitor and limit context size growth
  3. Cleanup Policies: Implement appropriate data retention policies
  4. Resource Monitoring: Monitor state storage resource usage

Security Considerations‚Äã

  1. Sensitive Data: Encrypt sensitive data in persistent state
  2. Access Control: Implement proper access controls for state data
  3. Audit Logging: Log all state access and modifications
  4. Data Retention: Follow compliance requirements for data retention
